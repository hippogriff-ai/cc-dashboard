# Continuity Ledger — cc-dashboard menubar conversion

## Goal
Convert `cc-dashboard` from a Python HTTP web dashboard into a self-contained macOS menu bar `.app` (Swift UI + TypeScript sidecar) that ships as a signed bundle, preserves every existing feature (ranked inbox, Restore view, Ghostty focus, resume/fork, open-in-IDE, staleness decay), adopts the UX design in `docs/ux-design/`, and adds Decision Log + Compost Heap projection registry + flashing status icon + Quiet mode + info-rich Session Detail panel. **Success criterion**: a single signed `cc-dashboard.app` bundle with no external runtime dependencies that replaces the browser-based dashboard.

## Constraints/Assumptions
- **Two-process bundle**: Swift menu bar shell spawns a Bun-compiled TypeScript sidecar over an ephemeral `127.0.0.1` HTTP port; sidecar binary embedded in `cc-dashboard.app/Contents/Resources/backend/`.
- **TypeScript sidecar (`backend/`)**: ESM only; zero runtime deps (`bun:*`, `node:*`, and own files); strict TS with `noUncheckedIndexedAccess`; throw `Error` instances; logging via `src/util/log.ts` (no bare `console.log` in `src/` except the port-announcement line in `server.ts`).
- **Swift app (`app/`)**: 4-space indent; SwiftUI/AppKit/local import order; `@MainActor` on UI view models; no force-unwraps in production; `os.Logger(subsystem: "dev.vcheval.cc-dashboard", category: ...)`; no SPM deps (vendored `KeyboardShortcuts` source files are the single exception, populated in Phase 5).
- **Build**: project file generated by xcodegen from `app/project.yml` (the `.xcodeproj` is gitignored); backend binary compiled via `bun build --compile`; all build/test routes through `Makefile` targets.
- **Stack**: macOS 14+, Swift 5.9, Bun 1.3.x runtime, XCTest, `bun test`.
- **Loop budget**: 12 iterations (per `/ralph-loop` invocation), not the 30 RALPH_PROMPT.md plans for. Critical path takes precedence over polish if budget exhausts.
- **Do NOT git commit or git push**: leave all changes unstaged for the user to review.

### Loop 26 — Smoke-test BLOCKER root cause identified
- **Root cause: `spawnSync` in `backend/src/util/git.ts`.** Each `gitInfo(cwd)` call makes THREE blocking subprocess calls (`git branch --show-current`, `git status --porcelain`, `git log -1 --pretty=...`) with 1500ms timeouts each. `loadRecentByRepo` (the `/api/recent` handler) calls `gitInfo` for EVERY cwd in the recent list. With even 10 recent repos, that's 30 synchronous git spawns × up to 1500ms = up to 45s blocking the Bun event loop — during which time NO other request can be served.
- **Why standalone works**: from the shell with `--port 18900`, only one curl request runs at a time. Each finishes before the next arrives. Even with slow `gitInfo`, the request completes before the user notices.
- **Why app-spawn hangs**: the Swift app polls `/api/live` every 2s AND `/api/recent` every 4s (Loop 16's PollingStore). On launch, the very first `/api/recent` call kicks off ~30 git spawns. While those are running, the Bun event loop can't accept the next `/api/live` request. URLSession's 5s timeout fires; PollingStore retries; the queue grows; everything looks dead.
- **Investigation steps that confirmed it**:
  - `sample` of the hung backend showed 760/768 samples in `kevent64` (Bun event loop waiting), with occasional `posix_spawn` calls.
  - The standalone binary with `--port 0` works perfectly (only proves Bun.serve isn't broken).
  - Pausing the parent Swift app via `kill -STOP` allowed an in-flight `/api/live` to complete (proving the backend CAN respond when not under polling pressure).
  - `git.ts` line 13: `spawnSync("git", ["-C", cwd, ...args], { encoding: "utf-8", timeout: timeoutMs })` is the smoking gun.
- **Fixes considered (NOT applied this loop)**:
  1. **Switch `gitInfo` to async `spawn`** with parallel awaits — would unblock the event loop. Maybe 30 lines of refactor in `git.ts`. Callers (`loadRecentByRepo`, `loadLiveSessions`, `buildPanel`) need to become async. Real fix.
  2. **Add a TTL cache** — memoize `gitInfo(cwd)` by cwd with say 30s TTL. Cuts spawn count by ~30× under steady polling. Smaller refactor (~15 lines) but doesn't help cold start.
  3. **Defer `gitInfo` to `/api/panel` only** — `/api/recent` and `/api/live` skip git, only the lazily-fetched panel includes git info. Largest behavior change but smallest code change to git.ts.
- **Codesign hypothesis (Loop 25 #2): ruled out**. The bun-compiled binary is already linker-signed (adhoc); re-signing fails "main executable failed strict validation" because of the JavaScriptCore data section. The existing signature is sufficient — codesign mismatch is NOT the cause of the hang.
- **Hardened runtime hypothesis (Loop 25 #1): ruled out**. Disabled and re-enabled; bug persists either way.
- **Files changed this loop:**
  - `app/project.yml` — restored `ENABLE_HARDENED_RUNTIME: YES`; postBuildScript codesign step attempted and reverted (commented why); ATS exception kept.
- **Tests**: 25/25 app + 74/74 backend still pass. No source-code fixes applied for the blocker — pure investigation.
